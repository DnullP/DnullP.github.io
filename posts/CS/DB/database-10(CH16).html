    overview

    expression tree

    We usually display an execution plan as a tree of relational-algebra operators, like:

    ΠA,B((σA=5(R))⋈(σB=6(S))) \Pi_{A,B}((\sigma_{A=5}(R)) \bowtie (\sigma_{B=6}(S))) ΠA,B​((σA=5​(R))⋈(σB=6​(S)))

    This expression can be depicted as a tree:

    basic steps

    • generating expressions that are logically equivalent to the given expression
    • annotating the resultant expressions in alternative ways to generate alternative query-evaluation plan
    • estimating the cost of each evaluating plan, and choosing the least one.

    We will discuss the first step detailly in the following sections.

    Above are the basic rules, but they just show the equivalence of two expressions. How to use them to optimize the query is another quesiton.

    Join Ordering

    A good ordering of join operations is important for reducing the size of temporary result

    So we should join the tables which might get a smaller result first.

    Enumeration of Equivalent Expressions

    Given an expression, we can calculate the closure of the expression by applying the equivalence rules.

    This is a time and space costly process, so we have some methods to optimize the process.

    • sharing the subexpressions which is not changed by the equivalence rules

    • reduce some unnecessary transformations

    Estimating Statistics of Expression Results

    We need to estimate the cost of a evaluation plan to choose the better one. Thus we need the statistics about the table (its metadata).

    join size estimate

    Assume that we have two relations rrr and sss and their attributes are AAA and BBB respectively.

    There are three cases:

    1. If the R∩S=∅R \cap S = \emptyR∩S=∅ , then the size of the join should just be 000.
    2. If the R∩SR \cap SR∩S is the key (it's unique), we can just use the key to join it. And the size should be nrn_rnr​ or nsn_sns​.

    choice of evaluation plan

    cost-based join-order selection

    We prefer to choose the join order that minimizes the total cost of the query.

    We call the result in an interesting sort order if the result is sorted on the join attribute, which is helpful for the next join operation.

    Cost-based Optimization of Equivalence Rules

    physical equivalence rule is to transform the logical operation into a physical operation like hahs join or sort merge join.

    Heuristics in Optimization

    Enumerate all the equivalent expressions is time and space costly, so we need to use some heuristics to optimize the process.

    For example, we can give a heuristic rule like:

    • Perform selection operation as early as possible

    We can learn it intuitively from that the selection always makes the result smaller.

    • Perform projection operations early

    We still have a lot of methods to optimize the query. One of the most important things is to trade-off between the query cost and the optimization cost.
