    Relation Database Design

    Features of the Good Relation Designs

    Decomposition

    • The aim of decomposing is to reduce the redundancy and avoid repetition of information.

    • lossy decomposition: cause the information loss

    When we decompose the relation RRR into R1R_1R1​ and R2R_2R2​, some information may be lost. Make an example:

    R(A‾,B,C‾,D) R(\underline{A},B,\underline{C},D) R(A​,B,C​,D)

    Here is a formula to check if a decomposition is lossless:

    R=R1∪R2R⇒DeR1,R2ΠR1(r)⋈ΠR2(r)=r R = R_1 \cup R_2 \\ R \xRightarrow{De} R_1 , R_2 \\ \Pi_{R_1}(r) \bowtie \Pi_{R_2}(r) = r R=R1​∪R2​RDe​R1​,R2​ΠR1​​(r)⋈ΠR2​​(r)=r

    The third line means that the natural join of the projection of R1R_1R1​ and R2R_2R2​ on r(R)r(R)r(R) is equal to r(R)r(R)r(R).

    Normalization Theory

    Basic funcion:

    • Decide whether it's a good form
    • Decompose the relation if not

    Before introducing the exact normalization, we should first redefine the primary key more formally.

    Superkey

    No two tuples in any legal instance of relation r(R)r(R)r(R) may have the same values on attribute set kkk.

    functional dependency

    Given r(R)r(R)r(R), α⊆R\alpha \subseteq Rα⊆R, β⊆R\beta \subseteq Rβ⊆R, and we can use the α→β\alpha \rightarrow \betaα→β to represent a functional dependency.

    This means for any two tuples t1t_1t1​ and t2t_2t2​ in r(R)r(R)r(R), if t1[α]=t2[α]t_1[\alpha] = t_2[\alpha]t1​[α]=t2​[α], then t1[β]=t2[β]t_1[\beta] = t_2[\beta]t1​[β]=t2​[β].

    And we say "α→β\alpha \rightarrow \betaα→β holds on r(R)r(R)r(R)", which means every instance of r(R)r(R)r(R) satisfies the functional dependency.

    So we can represent the primary key by the functional dependency:

    KKK is a superkey for r(R)r(R)r(R) if and only if K→RK \rightarrow RK→R holds on r(R)r(R)r(R).

    Additionally, we call the functional dependency which all the instance of r(R)r(R)r(R) satisfies as trivial functional dependency.

    Closure

    Closure is a common concept in computer science. Here we define it like below:

    Given r(R)r(R)r(R), α⊆R\alpha \subseteq Rα⊆R, β⊆R\beta \subseteq Rβ⊆R, and a set of functional dependencies FFF, we can use F+F^+F+ to represent the closure of FFF. All the functional dependencies in FFF hold on the r(R)r(R)r(R).

    The closure F+F^+F+ includes all the functional dependencies that can be inferred from the functional dependencies in FFF, as well as the dependencies in FFF itself.

    Additionally, we can also impose the concept of the closure onto the attribute set:

    A+A^+A+ is the set of all attributes that can be inferred by a given set of functional dependencies FFF.

    Lossless Decomposition with Functional Dependencies

    After given the formal definition of the functional dependency and closure, we can redefine the lossless decomposition:

    Given r(R)r(R)r(R), and the functional dependency set FFF, we can decompose losslessly r(R)r(R)r(R) into R1R_1R1​ and R2R_2R2​ if and only if the following condition holds:

    R1→R1∩R2→R1⊆F+R2→R1∩R2→R2⊆F+ R_1 \rightarrow R_1 \cap R_2 \rightarrow R_1 \subseteq F^+ \\ R_2 \rightarrow R_1 \cap R_2 \rightarrow R_2 \subseteq F^+ R1​→R1​∩R2​→R1​⊆F+R2​→R1​∩R2​→R2​⊆F+

    Normal Forms

    There are many kinds of normal forms, will just introduce some classic ones.

    Boyce-Codd Normal Form

    Given a closure of functional denpendency F+F^+F+, all the functional dependencies α→β\alpha \rightarrow \betaα→β in F+F^+F+ satisfies α⊆R\alpha \subseteq Rα⊆R, β⊆R\beta \subseteq Rβ⊆R.

    Then at least one of the following conditions holds:

    • α→β\alpha \rightarrow \betaα→β is trivial
    • α\alphaα is a superkey for r(R)r(R)r(R)

    If a functional dependency α→β\alpha \rightarrow \betaα→β is violated in the r(R)r(R)r(R), we can decompose the r(R)r(R)r(R) into two relations:

    • R1(α,β)R_1(\alpha, \beta)R1​(α,β)
    • R2(R−β,α)R_2(R - \beta, \alpha)R2​(R−β,α)

    BCNF can reduce the redundancy, and the error of update is also reduced.

    Dependency Preservation

    We will have more tables if we utilize the BCNF and decompose the relation. This will impair the performance of the query, primarily due to the increased need for join operations across multiple tables.

    When we query a tuple, we always want to get all the attributes that are uniquely determined by the primary key. But if we decompose them into two tables, such over-decomposition can lead to increased complexity in querying, as it may require joining multiple tables to retrieve all relevant attributes.

    Ensuring that all functionally dependent attributes remain within the same table to avoid complex join operations is known as dependency preservation.

    Update Anomalies

    Insertion Anomaly

    When inserting a tuple into a table, some attributes are not given due to some reason.

    We can certainly solve this by setting the nullable attribute to true. But under the relation design phase, we can just decompose the relation to solve this problem.

    Make a simple example:

    R(A‾,B,C,D) R(\underline{A}, B, C, D) R(A​,B,C,D)

    We can decompose it into R1(A‾,B,C)R_1(\underline{A}, B, C)R1​(A​,B,C) and R2(A‾,D)R_2(\underline{A}, D)R2​(A​,D).

    This allows us to just insert the (A,B,C)(A,B,C)(A,B,C) without the DDD.

    There are still issues of deletion anomalies and update anomalies to consider. However, these will not be discussed further.

    Thrid Normal Form

    After the BCNF and two kinds of purpose of decomposition. We will introduce a new kind of normal form: Third Normal Form.

    Given a closure of functional denpendency F+F^+F+, all the functional dependencies α→β\alpha \rightarrow \betaα→β in F+F^+F+ satisfies α⊆R\alpha \subseteq Rα⊆R, β⊆R\beta \subseteq Rβ⊆R.

    Then at least one of the following conditions holds:

    • α→β\alpha \rightarrow \betaα→β is trivial
    • α\alphaα is a superkey for r(R)r(R)r(R)
    • ∀A∈(β−α)\forall A \in (\beta - \alpha)∀A∈(β−α), AAA is contained in a candidate key for r(R)r(R)r(R)

    Third Normal Form is more relaxed than BCNF because it adds a new situation the normal form can hold.

    Comparision of 3NF and BCNF

    We have talked about the Dependency Preservation before, but we know that the BCNF doesn't always preserve the dependency:

    R(A‾,B,C,D) R(\underline{A}, B, C, D) R(A​,B,C,D)

    Given it the functional dependency {A→B,B→C}\{A \rightarrow B, B \rightarrow C\}{A→B,B→C}, we can decompose it into R1(A‾,B)R_1(\underline{A}, B)R1​(A​,B) and R2(B‾,C,D)R_2(\underline{B}, C, D)R2​(B​,C,D).

    But actually, we have an inferred functional dependency A→CA \rightarrow CA→C, and it's not preserved in the decomposition.

    And we can know that if AAA is the primary key, the BBB and CCC are the superkey, more, the candidate key.

    So we can preserve the original relation, and thanks to the third condition of the 3NF, BBB and CCC are contained in the candidate key. So the 3NF is not violated, and the dependency is preserved.

    However, if there is some missing value, we have to use Null or some default value to fill the table.

    Functional Theory

    In this part, we will introduce some basic concepts and theorems of the functional dependency.

    We know the partial order relation has three properties:

    • Reflexivity
    • Antisymmetry
    • Transitivity

    Functional dependency is similar but has some differences:

    • Reflexivity Rule: If α⊆R\alpha \subseteq Rα⊆R, and β⊆α\beta \subseteq \alphaβ⊆α, then α→β\alpha \rightarrow \betaα→β.

    • Augmentation Rule: If α→β\alpha \rightarrow \betaα→β and rrr is a set of attributes, then αr→βr\alpha r \rightarrow \beta rαr→βr.

    • Transitivity Rule: If α→β\alpha \rightarrow \betaα→β, and β→γ\beta \rightarrow \gammaβ→γ, then α→γ\alpha \rightarrow \gammaα→γ.

    Here we can see that the functional dependency is not antisymmetric.

    According to the three basic rules, we can get more conclusions:

    • Union Rule: If α→β\alpha \rightarrow \betaα→β, and α→γ\alpha \rightarrow \gammaα→γ, then α→βγ\alpha \rightarrow \beta \gammaα→βγ.

    • Decomposition Rule: If α→βγ\alpha \rightarrow \beta \gammaα→βγ, then α→β\alpha \rightarrow \betaα→β, and α→γ\alpha \rightarrow \gammaα→γ.

    • Pseudotransitivity Rule: If α→β\alpha \rightarrow \betaα→β, and γβ→δ\gamma \beta \rightarrow \deltaγβ→δ, then αγ→δ\alpha \gamma \rightarrow \deltaαγ→δ.

    Algorithm to Caculate the Closure

    This easy algorithm has been discussed in discrete mathematics. Here we just sketch it briefly.

    1. Initialize F+=FF^+ = FF+=F
    2. Repeat until no change to F+F^+F+:
      1. For each functional dependency α→β\alpha \rightarrow \betaα→β in F+F^+F+: If there is β→γ\beta \rightarrow \gammaβ→γ in F+F^+F+, and α→γ\alpha \rightarrow \gammaα→γ is not in F+F^+F+, add α→γ\alpha \rightarrow \gammaα→γ to F+F^+F+.

    And a similar algorithm can be applied to the attribute set AAA, too.

    canonical cover

    Extraneous

    If some attribute in the functional dependency can be removed without changing the closure, we call it extraneous.

    Suppose that α→β\alpha \rightarrow \betaα→β is a functional dependency.

    If we remove some attributes in the α\alphaα, it will weaken the constraint of the functional dependency. On the contrary, if we remove some attributes in the β\betaβ, it will strengthen the constraint.

    Usually, this action will change the closure of the attributes. But if the closure is not changed, we call the attribute extraneous.

    Canonical Cover

    A canonical cover must satisfy the following conditions:

    • No functional dependency in the FcF_cFc​ contains an extraneous attribute.
    • The left attributes of the functional dependency in the FcF_cFc​ are all unique.

    It means there is no such a situation that α→β\alpha \rightarrow \betaα→β and α→γ\alpha \rightarrow \gammaα→γ in the FcF_cFc​ according to the second condition.

    Algorithm to Caculate the Canonical Cover

    Here is an algorithm to caculate the canonical cover:

    1. Initialize Fc=FF_c = FFc​=F
    2. Repeat until no change to FcF_cFc​:
      1. Conbine all the α1→β1\alpha_1 \rightarrow \beta_1α1​→β1​ and α1→β2\alpha_1 \rightarrow \beta_2α1​→β2​ in the FcF_cFc​ into α1→β1β2\alpha_1 \rightarrow \beta_1 \beta_2α1​→β1​β2​. (according to the Union Rule)
      2. Find and remove the extraneous attributes in the α1\alpha_1α1​ and β1β2\beta_1 \beta_2β1​β2​.

    Dependency Preservation

    Here we will redefine the concept of dependency preservation based on the functional theory we talked above.

    We decompose the RRR into R1,R2…RnR_1, R_2\dots R_nR1​,R2​…Rn​, and the functional dependency of each RiR_iRi​ is FiF_iFi​. It's obviously that Fi⊆F+F_i \subseteq F^+Fi​⊆F+.

    Then we use F′F'F′ to represent the union of all the FiF_iFi​: F′=F1∪F2∪⋯∪FnF' = F_1 \cup F_2 \cup \dots \cup F_nF′=F1​∪F2​∪⋯∪Fn​.

    If F′+=F+F'^+ = F^+F′+=F+, we call the F′F'F′ implies the FFF, which means the decomposition is dependency preserved.

    Algorithm to Validate the Dependency Preservation

    Algorithm 1

    Just calculate the F′+F'^+F′+ and F+F^+F+, which will cost the exponential time.

    Algorithm 2

    1. Initialize result=αresult = \alpharesult=α
    2. Repeat until no change to resultresultresult:
      1. for each RiR_iRi​:
        1. t=(result∩Ri)+∩Rit = (result \cap R_i)^+ \cap R_it=(result∩Ri​)+∩Ri​
        2. result=result∪tresult = result \cup tresult=result∪t

    If resultresultresult contains all the attributes in β\betaβ, then the dependency α→β\alpha \rightarrow \betaα→β is preserved.

    One can comprehend the algorithm as follows:

    The algorithm begins with an initial set α\alphaα as the result. It iteratively refines this result by examining each relation RiR_iRi​ in the database. In each iteration, the algorithm calculates the closure of the intersection between α\alphaα and RiR_iRi​, denoted as ttt. This process essentially extracts the attributes that can be inferred within each RiR_iRi​ based on α\alphaα. The result is then updated by incorporating these inferred attributes. If, after these iterations, the resulting set contains all the attributes of β\betaβ, it indicates that the dependency α→β\alpha \rightarrow \betaα→β is maintained across the decomposed relations. This means that despite the decomposition, each part of the decomposed schema can collectively infer the complete set β\betaβ, thereby preserving the dependency.

    There are still algorithm to decompose the relation into the BCNF and 3NF, but we will not introduce them here.

    multi-valued dependency

    We use the notation below to represent the multi-valued dependency:

    α↠β \alpha \twoheadrightarrow \beta α↠β

    It means that for any two tuples t1t_1t1​ and t2t_2t2​ in r(R)r(R)r(R), if t1[α]=t2[α]t_1[\alpha] = t_2[\alpha]t1​[α]=t2​[α], then there exists:

    • t1[α]=t2[α]=t3[α]=t4[α]t_1[\alpha] = t_2[\alpha] = t_3[\alpha] = t_4[\alpha]t1​[α]=t2​[α]=t3​[α]=t4​[α]
    • t3[β]=t1[β]t_3[\beta] = t_1[\beta]t3​[β]=t1​[β]
    • t3[R−β]=t2[R−β]t_3[R - \beta] = t_2[R - \beta]t3​[R−β]=t2​[R−β]
    • t4[β]=t2[β]t_4[\beta] = t_2[\beta]t4​[β]=t2​[β]
    • t4[R−β]=t1[R−β]t_4[R - \beta] = t_1[R - \beta]t4​[R−β]=t1​[R−β]
